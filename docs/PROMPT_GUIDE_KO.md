# 프롬프트 튜닝 가이드

이 문서는 `src/lib/prompts.ts`를 어떻게 손봐야 출력 품질이 올라가는지 설명합니다.

---

## 1. 프롬프트 구조

현재 프롬프트는 4층 구조입니다.

1. 공통 규칙
2. 상황별 규칙
3. few-shot 예시
4. merchant profile + 실제 입력

### 공통 규칙
- 존댓말
- 사실 날조 금지
- 브랜드명 과다 반복 금지
- 세 답변 전략 다르게 생성
- JSON 외 텍스트 금지

### 상황별 규칙
- 리뷰
- 문의
- 클레임

### few-shot 예시
각 상황에 대해 좋은 출력 예시가 들어 있습니다.

### merchant profile
- 업종
- 브랜드명
- 기본 말투
- 요청 말투

---

## 2. 품질이 낮아지는 대표 원인

### 1) 프롬프트가 너무 추상적임
규칙만 많고 예시가 적으면 결과가 뻔해집니다.

### 2) 업종이 너무 넓음
스마트스토어/배달/카페/오프라인을 한 번에 다 잘하는 건 어렵습니다.

### 3) 실제 데이터가 없음
실제 고객 문장 없이 상상 예시만 넣으면 품질이 안 올라갑니다.

### 4) 전략 분리가 약함
세 답변이 이름만 다르고 문장이 비슷해지는 문제가 생깁니다.

---

## 3. 가장 효과 좋은 개선법

### A. 업종 1개만 먼저 집중
예를 들어 스마트스토어만 집중하면 아래 상황 few-shot을 늘리세요.

- 배송지연
- 재입고
- 단순변심 환불
- 오배송
- 파손
- 포장 불만
- 저별점 리뷰
- 악성 리뷰

### B. 금지 표현을 명시
예:
- “100%”
- “절대 문제 없습니다”
- “고객님이 잘못 이해하신 것 같습니다”
- “저희 책임은 아닙니다만”
- “규정상 안 됩니다”만 던지는 식의 차가운 문장

### C. 길이 범위를 더 좁힘
지금은 복붙 즉시 사용 중심입니다.  
너무 길게 나오면 현업에서는 안 씁니다.

실전에서는 다음 정도가 좋습니다.

- 리뷰 답변: 60~110자
- 문의 답변: 45~100자
- 클레임 답변: 70~130자

### D. 실제 베스트 문장 넣기
네가 직접 “이건 사람이 쓴 것 같고 실제로 쓸만하다” 싶은 문장을 넣어야 합니다.

---

## 4. 추천 튜닝 루프

1. 실제 문의/리뷰/클레임 30개 수집
2. 현재 앱에 넣어 생성
3. 별로인 결과 저장
4. 왜 별로인지 유형화
5. 프롬프트 수정
6. 다시 생성
7. 반복

---

## 5. 어디를 고치면 되는가

### 공통 규칙
`buildCommonInstructions()`

### 리뷰 규칙
`buildReviewInstructions()`

### 문의 규칙
`buildInquiryInstructions()`

### 클레임 규칙
`buildComplaintInstructions()`

## 6. 세 답변을 진짜 다르게 만드는 방법

프롬프트에서 다음을 더 강하게 요구하면 좋습니다.

- 첫 문장 구조를 다르게
- 정보 순서를 다르게
- 마무리를 다르게
- 감정 완화 정도를 다르게

예:
- 따뜻한형: 감정 완화 우선
- 균형형: 핵심 사실 + 정중한 마무리
- 원칙형: 짧고 기준 중심

---

## 7. 품질 guard 로직

`src/lib/generation-service.ts`에 간단한 품질 guard가 들어 있습니다.

- 중복 텍스트 여부
- 시작 문장 유사도
- 토큰 중복도

너무 비슷하면 한 번 더 재생성합니다.

이 guard는 완벽하진 않지만 “세 개가 사실상 똑같은 문제”를 줄이는 데 도움 됩니다.

---

## 8. 비용 대비 품질 조절

### 더 싸게
- 모델 유지
- `OPENAI_REASONING_EFFORT=low`

### 더 신중하게
- `OPENAI_REASONING_EFFORT=medium` 또는 `high`

### 더 안정적으로
- 더 좋은 few-shot
- 업종 집중
- 금지 표현 강화

대부분의 경우 **좋은 예시 추가가 모델 업그레이드보다 효율이 좋습니다.**
